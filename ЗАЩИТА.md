# Текст выступления на защите дипломного проекта

**Тема:** «TestCenter — веб-платформа тестирования знаний студентов»  
**Организация:** Актюбинский высший политехнический колледж (АВПК)

---

## Приветствие

Уважаемые члены аттестационной комиссии, разрешите представить вашему вниманию дипломный проект на тему «TestCenter — веб-платформа тестирования знаний студентов Актюбинского высшего политехнического колледжа».

---

## Актуальность

В современных условиях цифровизации образования традиционное бумажное тестирование имеет ряд существенных недостатков:

- Высокие временны́е затраты преподавателей на составление, проведение и проверку тестов вручную
- Отсутствие мгновенной обратной связи для студента — результат становится известен спустя дни
- Невозможность хранить и анализировать историю успеваемости в удобном виде
- Бумажные бланки теряются, портятся, не поддаются систематизации
- Отсутствие подтверждающих документов — сертификатов — о прохождении аттестации

Данный проект решает все перечисленные проблемы, предоставляя колледжу полноценную цифровую платформу, доступную с любого устройства через браузер без установки дополнительного программного обеспечения.

---

## Цель и задачи

**Цель проекта** — разработать веб-приложение для автоматизации процесса тестирования знаний студентов с поддержкой трёх ролей пользователей: студент, преподаватель и администратор.

**Задачи:**

1. Спроектировать ролевую систему разграничения доступа
2. Реализовать конструктор тестов для преподавателей с поддержкой трёх типов вопросов
3. Разработать интерактивный интерфейс прохождения теста с обратным отсчётом времени
4. Реализовать автоматическую проверку результатов и выставление оценки
5. Создать систему генерации PDF-сертификатов при успешном прохождении
6. Обеспечить хранение данных на стороне клиента без необходимости серверной инфраструктуры
7. Развернуть приложение на бесплатном хостинге GitHub Pages

## Технологический стек

### React 19 + TypeScript

React — наиболее популярная библиотека для построения пользовательских интерфейсов, использующая компонентный подход: интерфейс разбивается на независимые переиспользуемые части — компоненты. TypeScript добавляет статическую типизацию поверх JavaScript, что исключает целый класс ошибок ещё на этапе написания кода и делает его самодокументируемым.

### Vite 8

Современный инструмент сборки проекта. Обеспечивает мгновенный запуск сервера разработки и быструю итоговую сборку за счёт нативной поддержки ES-модулей браузером. В отличие от устаревшего Webpack, Vite не перебирает весь проект при каждом изменении файла — только нужный модуль.

### Tailwind CSS 3

Utility-first CSS-фреймворк. Позволяет задавать стили непосредственно в JSX-разметке с помощью коротких классов (`p-4`, `text-blue-600`, `rounded-xl`) без написания отдельных CSS-файлов. Обеспечивает единообразие отступов, цветов и типографики по всему приложению и автоматически удаляет неиспользуемые стили при сборке.

### Radix UI + shadcn/ui

Radix UI — библиотека «безголовых» компонентов с полной поддержкой доступности: модальные окна, выпадающие списки, вкладки корректно обрабатывают навигацию с клавиатуры и скринридеры. shadcn/ui предоставляет готовые стилизованные компоненты поверх Radix UI — Button, Card, Input, Badge, Select, Progress и другие, ускоряя разработку интерфейса.

### React Router v7

Библиотека клиентской маршрутизации. Обеспечивает навигацию между страницами без перезагрузки браузера. Поддерживает вложенные маршруты и параметры URL (`/student/test/:id`). В проекте используется компонент `ProtectedRoute`, который проверяет роль пользователя перед отображением страницы.

### Zustand

Минималистичная библиотека для глобального состояния приложения. Выбрана вместо Redux за отсутствие шаблонного кода — весь стор аутентификации умещается в 60 строк против нескольких сотен в Redux. Хранит текущего пользователя и предоставляет методы `signIn`, `signUp`, `signOut`, `loadSession`.

### React Hook Form + Zod

React Hook Form обеспечивает высокопроизводительную работу с формами без лишних перерисовок компонентов. Zod — библиотека валидации на TypeScript: описывается схема данных формы, и все ошибки валидации (некорректный email, пароли не совпадают, поле пустое) обрабатываются автоматически.

### Lucide React

Набор из более чем 1000 SVG-иконок в виде React-компонентов. Используется повсеместно в интерфейсе — иконки навигации, статистики, кнопок действий.

### localStorage

Встроенный браузерный механизм хранения данных в формате JSON. Позволяет хранить пользователей, тесты, результаты и сертификаты без какой-либо серверной инфраструктуры. Объём хранилища — до 5–10 МБ в зависимости от браузера, чего достаточно для сотен тестов и тысяч попыток.

### GitHub Pages

Бесплатный хостинг статических сайтов от GitHub. Деплой производится одной командой `npm run deploy` через пакет `gh-pages`, который собирает проект и публикует результат в отдельную ветку репозитория.

## Архитектура системы

Приложение построено по архитектуре **SPA (Single Page Application)** — одностраничное приложение, где навигация между разделами происходит без перезагрузки страницы. Браузер единожды загружает HTML, CSS и JavaScript, после чего все переходы обрабатываются клиентским кодом.

### Структура проекта

```
src/
├── components/
│   ├── ui/           — базовые компоненты: Button, Card, Input, Badge, Select и др.
│   ├── auth/         — ProtectedRoute (защита маршрутов по роли)
│   ├── layout/       — Header, Sidebar, PageLayout
│   ├── dashboard/    — StatsCard
│   ├── test/         — TestCard, Timer
│   └── certificate/  — CertificatePDF
├── pages/
│   ├── student/      — Dashboard, TestList, TakeTest, Results, Certificates
│   ├── teacher/      — Dashboard, CreateTest, EditTest, TestResults, Students
│   └── admin/        — Dashboard, Users, Groups, Subjects
├── store/
│   └── authStore.ts  — глобальное состояние аутентификации (Zustand)
├── hooks/
│   ├── useTests.ts
│   ├── useAttempts.ts
│   └── useCertificates.ts
├── lib/
│   ├── db.ts         — слой работы с localStorage

    └── index.ts      — TypeScript-интерфейсы: User, Test, Question, Attempt, Certificate
```

### Слой данных (db.ts)

Ключевое архитектурное решение проекта. Модуль реализует паттерн «репозиторий» поверх `localStorage`: все операции чтения и записи инкапсулированы в объект `db` с именованными методами — `getTests()`, `createAttempt()`, `createCertificate()` и т.д. Остальной код приложения не работает с `localStorage` напрямую — только через этот слой. Это означает, что при необходимости перехода на реальную базу данных достаточно заменить реализацию `db.ts`, не трогая ни одну страницу или компонент.

При первом запуске вызывается `initDb()`, который заполняет хранилище начальными данными: группами, предметами, тестами с вопросами и тестовыми аккаунтами.

### Защита маршрутов

Компонент `ProtectedRoute` оборачивает каждую защищённую страницу. При обращении к маршруту он проверяет два условия: авторизован ли пользователь и соответствует ли его роль требованиям маршрута. Если нет — перенаправляет на страницу входа или на главную. Например, страница `/admin/users` доступна только с ролью `admin`. Преподаватель или студент, вручную набравший этот адрес, будет перенаправлен.

## Роли пользователей и функционал

Система поддерживает три роли, каждая из которых видит свой изолированный интерфейс.

### Студент

- **Дашборд** — сводная статистика: количество пройденных тестов, средний балл, количество сертификатов, количество успешно сданных
- **Список тестов** — просмотр доступных активных тестов с поиском по названию и фильтрацией по предмету
- **Прохождение теста** — пошаговый интерфейс с отображением прогресса, обратным таймером, поддержкой трёх типов вопросов, точечной навигацией между вопросами
- **Результаты** — история всех попыток с баллом, словесной оценкой и датой
- **Сертификаты** — список полученных сертификатов с возможностью сохранить PDF

### Преподаватель

- **Дашборд** — список созданных тестов, общее число попыток по всем тестам, средний балл
- **Создание теста** — форма с параметрами (название, предмет, лимит времени, проходной балл, число разрешённых попыток) и конструктором вопросов с вариантами ответов
- **Редактирование теста** — изменение параметров и вопросов уже созданного теста
- **Результаты теста** — таблица попыток всех студентов по выбранному тесту
- **Список студентов** — просмотр зарегистрированных в системе студентов

### Администратор

- **Дашборд** — сводная статистика по всему колледжу: пользователи, тесты, сертификаты, группы
- **Пользователи** — управление учётными записями
- **Группы** — создание и редактирование учебных групп
- **Предметы** — управление дисциплинами и привязкой к преподавателям

## Типы вопросов и система оценивания

Конструктор тестов поддерживает три типа вопросов.

**Один правильный ответ (single).** Студент выбирает один вариант из нескольких предложенных. Балл начисляется только при точном совпадении.

**Несколько правильных ответов (multiple).** Студент выбирает все верные варианты. Балл начисляется только при полном совпадении набора выбранных ответов с набором правильных — частичного зачёта нет. Это соответствует принципу строгой проверки знаний.

**Верно / Неверно (truefalse).** Формулируется утверждение, студент выбирает один из двух вариантов.

**Алгоритм подсчёта результата:**

> Итоговый балл = (сумма баллов за правильные ответы / максимально возможная сумма) × 100%

Каждый вопрос имеет свой вес в баллах, задаваемый преподавателем при создании. Проходной балл также задаётся индивидуально для каждого теста.

**Шкала оценок:**

| Балл | Оценка |
|---|---|
| 90–100% | Отлично |
| 75–89% | Хорошо |
| 60–74% | Удовлетворительно |
| Ниже 60% | Неудовлетворительно |

## Таймер и автозавершение теста

В процессе прохождения теста в шапке отображается обратный таймер. Компонент `Timer` реализован через `useEffect` и `setInterval` — каждую секунду счётчик уменьшается на единицу. При остатке менее 60 секунд таймер меняет цвет на красный — визуальное предупреждение студенту.

Когда время истекает, метод `handleExpire` вызывается автоматически: рассчитывается балл по уже данным ответам и результат сохраняется. Студент не может продолжать отвечать после истечения времени — это предотвращает недобросовестное прохождение.

## Генерация сертификатов

При успешном прохождении теста — когда набранный балл не ниже проходного — в базе данных автоматически создаётся запись сертификата с уникальным кодом верификации в формате `АВПК-[год]-[5-значный номер]`.

Сертификат генерируется на основе HTML/CSS-шаблона, который открывается в отдельном окне браузера. Для корректного отображения кириллического текста подключается шрифт Roboto через Google Fonts. Диалог печати открывается автоматически — студент выбирает «Сохранить как PDF» в принтере операционной системы.

Сертификат оформлен в формате A4 альбомной ориентации и содержит: полное название колледжа на русском и казахском языках, ФИО студента, название предмета и теста, дату выдачи, набранный балл и оценку, уникальный код верификации, место для подписи преподавателя.

## Хранение данных

Принципиальное архитектурное решение — отказ от серверной базы данных в пользу браузерного `localStorage`. Причины: приложение размещается на GitHub Pages как статический сайт без арендованного сервера, исключаются расходы на хостинг и обслуживание, отсутствует зависимость от внешних сервисов.

Данные хранятся в девяти ключах localStorage в формате JSON: `tc_users`, `tc_groups`, `tc_subjects`, `tc_tests`, `tc_questions`, `tc_answers`, `tc_attempts`, `tc_certificates`, `tc_session`.

При первом запуске система автоматически заполняется начальными данными: 2 группы, 3 предмета, 3 теста с пятью вопросами каждый — итого 15 вопросов по дисциплинам колледжа, а также 3 тестовых аккаунта для демонстрации.

Ограничение данного подхода: данные хранятся локально в браузере конкретного устройства. Для сценариев, где данные должны быть общими между устройствами, потребуется переход на серверную базу данных — это определено как направление дальнейшего развития.

## Безопасность и разграничение доступа

Каждая страница обёрнута в компонент `ProtectedRoute`, который при обращении к маршруту проверяет: авторизован ли пользователь и соответствует ли его роль требованиям маршрута. Несоответствие — немедленное перенаправление.

Идентификатор текущего пользователя хранится в ключе `tc_session`. При закрытии и повторном открытии браузера сессия восстанавливается автоматически. При нажатии «Выйти» ключ удаляется.

Все формы валидируются с помощью библиотеки Zod до обработки данных. При регистрации проверяются: минимальная длина ФИО (3 символа), корректность формата email, минимальная длина пароля (6 символов), совпадение пароля и подтверждения. Ошибки отображаются непосредственно под соответствующим полем.

## Процесс разработки

Разработка велась в среде **WebStorm** (JetBrains) на операционной системе **Windows 11**.

Инструменты: **Git и GitHub** для контроля версий и хранения кода, **npm** для управления пакетами, **Vite Dev Server** с горячей перезагрузкой для мгновенного отображения изменений, **TypeScript Compiler** для проверки типов в реальном времени.

**Этапы разработки:**

1. Проектирование структуры данных и TypeScript-интерфейсов
2. Создание слоя данных `db.ts` с функцией инициализации и тестовыми данными
3. Реализация системы аутентификации и защиты маршрутов
4. Разработка модуля прохождения теста с таймером и подсчётом баллов
5. Реализация конструктора тестов для преподавателя
6. Создание дашбордов для каждой из трёх ролей
7. Разработка генерации PDF-сертификатов
8. UI-полировка: адаптивность, состояния загрузки, визуальные переходы, единообразие карточек
9. Деплой на GitHub Pages

Деплой производится командой `npm run deploy`, которая последовательно выполняет проверку типов TypeScript, сборку Vite и публикацию папки `dist` на GitHub Pages.

## Вероятные вопросы комиссии

**Почему не использовали готовую платформу, например Moodle?**

Moodle требует собственного сервера, системного администратора и значительных ресурсов на настройку и поддержку. TestCenter — лёгкое решение, точно заточенное под нужды АВПК, разворачиваемое без инфраструктуры за несколько минут. Кроме того, разработка собственной системы — это полноценная демонстрация профессиональных компетенций, что и является целью дипломного проекта.

**Почему данные хранятся в localStorage, а не в базе данных?**

Для дипломного проекта это оптимальное решение: нет затрат на хостинг сервера, нет зависимости от внешних сервисов, приложение работает полностью автономно. Архитектура намеренно спроектирована так, чтобы слой `db.ts` при необходимости можно было заменить на любую реальную БД — Firebase, PostgreSQL — без изменения ни одной страницы или компонента.

**Как обеспечивается безопасность?**

Маршруты защищены проверкой роли на уровне `ProtectedRoute`. Валидация данных выполняется через схемы Zod. В рамках дипломного проекта пароли хранятся в открытом виде — для реального продакшена следующим шагом является хэширование через bcrypt.

**Сколько студентов может одновременно работать с системой?**

Поскольку каждый пользователь работает со своим браузером и своим localStorage, нагрузка не централизована и не ограничена. Каждый студент — это изолированная копия данных в его браузере.

**Какие технологии были для вас новыми в процессе разработки?**

TypeScript как строго типизированный язык программирования, Zustand для управления глобальным состоянием, работа с React Router v7 и его системой вложенных защищённых маршрутов — всё это изучалось и применялось непосредственно в ходе разработки дипломного проекта.

**Почему выбран React, а не другой фреймворк?**

React — наиболее востребованная на рынке труда библиотека для построения интерфейсов. По данным опроса Stack Overflow 2024 года, React используют более 39% разработчиков. Выбор React обеспечивает применимость полученных навыков в профессиональной деятельности.
